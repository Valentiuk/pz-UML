# Практичне заняття: pz-UML
## Проєктування інтелектуальної системи керування паркінгом "SmartPark"

Предметна область: Інформаційна система автоматизації паркувального простору. Система забезпечує безконтактний в'їзд завдяки розпізнаванню автономерів, контроль заповнюваності, гнучку тарифікацію та інтеграцію з зовнішніми платіжними сервісами.
---

### 1. Діаграма варіантів використання (Use Case Diagram)
Мета: Визначення меж системи, основних функціональних можливостей та ролей користувачів (акторів), які з нею взаємодіють.

```mermaid
flowchart LR
    %% Актори
    Driver((Водій))
    VIP((Резидент))
    Guest((Гість))
    Admin((Оператор))
    Camera((Камера OCR))
    Bank((Платіжний шлюз))

    %% Наслідування акторів
    VIP -.-> Driver
    Guest -.-> Driver

    %% Межі системи
    subgraph Smart Parking System
        direction TB
        UC1([Реєстрація в'їзду])
        UC2([Розпізнавання номера])
        UC4([Видача разового талона])
        UC5([Оплата послуг])
        UC6([Безготівкова транзакція])
        UC7([Виїзд з паркінгу])
        UC8([Моніторинг вільних місць])
        UC9([Керування тарифами])
    end

    %% Взаємодія акторів з прецедентами
    Driver --- UC1
    Driver --- UC5
    Driver --- UC7
    
    Guest --- UC4
    
    Admin --- UC8
    Admin --- UC9
    
    Camera --- UC2
    Bank --- UC6

    %% Відношення між прецедентами
    UC1 -. "<<include>>" .-> UC2
    UC5 -. "<<include>>" .-> UC6
    UC4 -. "<<extend>>" .-> UC1
```

Аналіз: Діаграма демонструє, що система обслуговує два типи водіїв (резидентів з абонементами та гостей). Виділено ключові зовнішні модулі (камера та банк), які автономно виконують частину роботи, розвантажуючи адміністратора.

---

### 2. Діаграма послідовності (Sequence Diagram)
Сценарій: Процес в'їзду автомобіля на територію, розпізнавання номера та взаємодія з апаратним забезпеченням.

```mermaid
sequenceDiagram
    autonumber
    actor Car as Автомобіль
    participant InCam as Камера В'їзду
    participant Backend as Сервер Паркінгу
    participant DB as База Даних
    participant Gate as Шлагбаум

    rect rgb(235, 245, 255)
    note right of Car: ФАЗА В'ЇЗДУ
    Car->>InCam: Під'їзд до стоп-лінії
    InCam->>InCam: Розпізнавання номера (OCR)
    InCam->>Backend: Передача номера (AA1111BB)
    Backend->>DB: Запит наявності броні/абонемента
    
    alt Абонемент знайдено
        DB-->>Backend: Відповідь (Статус: Резидент)
        Backend->>DB: Створення сесії (Час IN)
        Backend->>Gate: Команда: Відкрити
        Gate-->>Car: Шлагбаум відкрито
        Car->>Gate: Проїзд (перетин магнітної петлі)
        Gate->>Backend: Сигнал: Авто проїхало
        Backend->>Gate: Команда: Закрити
    else Номер не знайдено
        DB-->>Backend: Відповідь (Статус: Невідомий)
        Backend-->>InCam: Відмова (Потрібен квиток)
        InCam-->>Car: Виведення повідомлення "Візьміть талон"
    end
    end
```

Аналіз: Діаграма чітко показує хронологію викликів. Сервер виступає центральним контролером, який приймає рішення на основі даних з БД і лише потім відправляє команду на відкриття шлагбаума.

### 3. Діаграма діяльності (Activity Diagram)
Мета: Візуалізація бізнес-логіки та алгоритму розрахунку клієнта за паркування.

```mermaid
stateDiagram-v2
    [*] --> ПідхідДоТерміналу
    
    ПідхідДоТерміналу --> ВведенняНомераАвто
    ВведенняНомераАвто --> ЗапитДоБД
    ЗапитДоБД --> РозрахунокСуми
    
    state fork_pay <<fork>>
    РозрахунокСуми --> fork_pay
    
    fork_pay --> ОплатаКарткою
    fork_pay --> ОплатаГотівкою
    
    state join_pay <<join>>
    ОплатаКарткою --> ВалідаціяТранзакції
    ОплатаГотівкою --> ВалідаціяКупюр
    
    ВалідаціяТранзакції --> join_pay
    ВалідаціяКупюр --> join_pay
    
    join_pay --> ПеревіркаСтатусу
    
    state ПеревіркаСтатусу <<choice>>
    ПеревіркаСтатусу --> Помилка : Відхилено банком
    Помилка --> ПідхідДоТерміналу : Спробувати ще раз
    
    ПеревіркаСтатусу --> ОплатаУспішна : Транзакція ОК
    ОплатаУспішна --> ДрукЧеку
    ДрукЧеку --> ЗапускГрейсПеріоду
    
    ЗапускГрейсПеріоду --> [*]
    note right of ЗапускГрейсПеріоду: Клієнт має 15 хв на виїзд
```

Аналіз: Відображено паралельні процеси вибору методу оплати (`fork`/`join`) та умовне розгалуження на випадок збою транзакції. Логіка завершується стартом "грейс-періоду", що пов'язує цю діаграму з подальшим процесом виїзду.

---

### Висновки
Побудовані діаграми повністю описують архітектуру поведінки розроблюваної системи:
1. Use Case окреслила загальні функціональні вимоги та користувачів.
2. Sequence деталізувала технічну взаємодію між сервером, базою даних та апаратним забезпеченням у часі.
3. Activity дозволила продумати алгоритмічні кроки та обробку помилок у критичному бізнес-процесі (оплата).

Діаграми пов'язані між собою єдиною предметною областю та виконані з використанням підходу Diagrams as Code (Mermaid.js).
